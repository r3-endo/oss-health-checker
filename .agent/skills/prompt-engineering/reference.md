# Prompt Engineering 詳細方針

## 目的
- LLMのパフォーマンス、信頼性、制御性を高める
- 期待する出力の一貫性を向上させる

## 進め方の原則
- 一度に一つの質問だけを投げる
- 可能なら選択式の質問を優先する
- 2〜3の設計方針を提示してから結論を出す
- 必要なら、スコープを明示的に狭める

## 設計のチェック観点
- 目的: 何を達成したいかが一文で言える
- 制約: 文字数、形式、トーン、禁止事項が明確
- 入力: 受け取る情報と前提条件が明確
- 出力: 期待する構造と形式が具体的
- 評価: 何を持って良い出力とするかが定義されている

## テンプレート（簡易）

### 目的確認
- 何を解決したいか？
- 誰が使うか？
- 成功の条件は何か？

### 代替案提示（2〜3）
- 案A: シンプル重視（制約は少なめ）
- 案B: 制御重視（フォーマット厳格）
- 案C: 安全重視（チェックリスト付き）

## よくあるアンチパターン
- 指示が抽象的すぎて再現性がない
- 目的と出力フォーマットが噛み合っていない
- 例示（few-shot）が目的に合っていない
- 制約を増やしすぎて柔軟性を失う

## 評価観点（簡易）
- 再現性: 同条件で同品質が出るか
- 制御性: 望まない出力を抑制できているか
- 簡潔性: 指示が必要十分か
- 一貫性: 形式・トーンが安定しているか
